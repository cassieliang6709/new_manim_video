Editorial Rebuild

Rendered. Not just generated.

new_manim_video is an AI pipeline that converts broad natural-language prompts into actual animation videos. The work is not interesting because a model can draft code. It is interesting because the code has to survive execution, recover from failure, and return a real MP4.

Project stance Most AI pages try to look polished first. This one is intentionally more like a magazine spread: oversized type, sharp section breaks, and the actual output videos placed directly inside the story.
Why it matters

A system like this needs structure around the model: retrieval, audit, sandboxing, error compression, retries, and a fallback path. Otherwise it is just a fragile demo.

01
Problem

The hard part is after the prompt.

Users describe visual ideas loosely: “show Fourier series,” “explain gradient descent,” “animate bubble sort.” Those requests carry intent, not implementation. The system still has to infer scene structure, pick the right Manim constructs, avoid unsafe code, and render a usable result.

That means the system is judged by what happens after generation. If the render fails, the experience is broken. If the code is unsafe, the architecture is broken. If the retry loop spirals forever, the product is broken.

02
System

A multi-stage loop, not one prompt call.

The pipeline retrieves useful examples from prior successes, generates code, audits that code, executes rendering inside Docker, captures failures, compresses the traceback, injects better context, and retries with tighter guidance.

Each stage exists because the failure modes differ. Retrieval helps with grounding. Audit helps with safety. Sandbox helps with execution boundaries. Retry helps with recovery. Fallback helps when recovery still fails.

03
Execution logic

Why the architecture is shaped this way.

Rendering model-written code is a different category of problem from generating static text. The code must be isolated. The error signal must be short enough to be useful. Retries must be bounded. And when the main path fails, the user should still get a meaningful result instead of silence.

In other words: the architecture is there to turn an unreliable model into a more reliable product loop.
