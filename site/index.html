Editorial Case Study

From plain English to rendered animation.

new_manim_video is an AI system that turns natural-language prompts into real Manim videos. The hard part was not generating code. The hard part was making the system survive ambiguity, failed renders, unsafe generations, and broken APIs long enough to ship an actual media artifact.

This page is intentionally quiet. It is not a startup landing page. It is a compact project document for showing what the system does, why the engineering was hard, and what the outputs actually look like.

Story

What I was trying to build

Most LLM demos stop at generated code. This project goes one step further: the code needs to execute, render, and produce a usable video file. That changes the problem completely. Prompt quality still matters, but orchestration, auditing, sandboxing, and recovery matter more.

A user can ask for something vague like “show the Fourier series approximation of a square wave” or “visualize gradient descent.” Those prompts encode intent, not implementation. The system has to infer scene structure, choose APIs, avoid unsafe code, handle render errors, and keep retrying with better context instead of collapsing after the first failure.

The goal was not: get a nice-looking code snippet from a model.
The goal was: return a playable MP4, even when the first attempt was wrong.
System

How the pipeline works

The repo is organized as a workflow, not a single prompt call. Each stage has a narrow job, and each one exists because the failure modes are different.

Before rendering

The system retrieves useful examples from prior successful runs, generates Manim code with an LLM, then checks that code with both AST-based security rules and semantic review.

After failure

If rendering breaks, the traceback is compressed into a smaller signal, relevant API hints are injected, and the next generation attempt is more targeted rather than purely random.

Execution boundary

Rendering runs inside Docker so generated code is constrained. That matters because this system executes model-written code, not just text.

Fallback behavior

Retries are bounded. If the main loop cannot recover, the pipeline still aims to produce a safe fallback scene rather than leaving the user with an empty failure.

Stack

The main ingredients

The implementation combines several small systems that each solve a specific problem in the loop:

  • Python service layer for the app, orchestration, and reusable entry points.
  • LangGraph for multi-stage workflow routing and retry logic.
  • Gemini models for generation, diagnosis, and judging.
  • Docker for isolated Manim rendering.
  • Sentence Transformers retrieval over past successful runs.
  • Optional Google Drive upload and MCP interface for integration.

The source code and run paths live in the repository, especially app.py, service_api.py, orchestrator.py, executor.py, and the template library under animation_templates/.
